# How to Solve Find the K-th Character in String Game I

This problem presents a deceptively simple game where Alice starts with the string `"a"` and repeatedly transforms it by shifting each character to the next letter in the alphabet, then appending the result to the original string. Given a positive integer `k`, we need to find the k-th character (1-indexed) in the infinite string generated by this process. The challenge lies in efficiently determining this character without actually constructing the potentially massive string.

What makes this problem interesting is that while it appears to be about string manipulation, the optimal solution uses mathematical reasoning about binary representations. The infinite growth pattern reveals a hidden structure that allows us to compute the answer in O(log k) time instead of simulating the process.

## Visual Walkthrough

Let's trace through the process with a small example to build intuition. We'll find the 5th character (k=5) step by step.

**Initial state:** word = "a"

**Operation 1:**

- Take "a", shift each character: "a" → "b"
- Append to original: "a" + "b" = "ab"
- New word: "ab"

**Operation 2:**

- Take "ab", shift each character: "a"→"b", "b"→"c" → "bc"
- Append to original: "ab" + "bc" = "abbc"
- New word: "abbc"

**Operation 3:**

- Take "abbc", shift: "a"→"b", "b"→"c", "b"→"c", "c"→"d" → "bccd"
- Append: "abbc" + "bccd" = "abbcbccd"
- New word: "abbcbccd"

Now let's examine the string after 3 operations: "a b b c b c c d"
Positions: 1 2 3 4 5 6 7 8

The 5th character is 'b'. But notice something interesting: the string length doubles each operation (1 → 2 → 4 → 8 → 16...). This exponential growth means brute force simulation quickly becomes impossible for large k.

More importantly, observe the pattern: each character's value depends on its position in a binary tree-like structure. The first character is always 'a'. For any position i, if we repeatedly halve i until we reach 1, each time we take the "right half" (i > length/2), we're effectively applying a shift operation.

## Brute Force Approach

The most straightforward approach is to simulate the game exactly as described:

1. Start with word = "a"
2. While the length of word is less than k:
   - Generate shifted version by adding 1 to each character's ASCII value
   - Append shifted version to original word
3. Return the k-th character (1-indexed)

```python
# Time: O(k) | Space: O(k) - Too slow for large k!
def bruteForce(k):
    word = "a"

    while len(word) < k:
        # Generate shifted version
        shifted = ""
        for ch in word:
            # Shift to next character, handle 'z' -> 'a' if needed
            # (Though problem guarantees we won't reach 'z' for given constraints)
            shifted += chr(ord(ch) + 1)

        # Append shifted version
        word += shifted

    # Return k-th character (1-indexed)
    return word[k-1]
```

**Why this fails:** The string length grows exponentially (1, 2, 4, 8, 16, 32...). For k up to 10^9 (a typical constraint), we'd need to build a string of length at least k, which requires O(k) time and space. This is completely infeasible. We need a solution that doesn't depend on the string length.

## Optimal Solution

The key insight is recognizing the binary structure of the problem. Let's analyze the pattern:

1. After 0 operations: "a" (length = 1)
2. After 1 operation: "ab" (length = 2)
3. After 2 operations: "abbc" (length = 4)
4. After 3 operations: "abbcbccd" (length = 8)

Notice that at each step, the second half is exactly the first half with each character incremented by 1. This creates a recursive structure:

- The first character is always 'a'
- For any position i in [1, n] where n is the current length:
  - If i is in the first half (i ≤ n/2), it's the same as position i in the previous iteration
  - If i is in the second half (i > n/2), it's equal to the character at position (i - n/2) in the previous iteration, plus 1

We can think of this as a binary tree where:

- Root is 'a' at position 1
- Left child of a node is the same character
- Right child is the character + 1

To find the k-th character:

1. Start with base character 'a'
2. While k > 1:
   - Find the smallest power of 2 that is ≥ k
   - If k is in the second half of this power of 2, subtract half and increment character
   - Otherwise, keep character the same
   - Update k to its position in the previous iteration
3. Return the final character

Equivalently, we can track how many times we take the "right child" in the binary tree. Each time we take the right child, we increment the character by 1. The number of right-child traversals equals the number of 1's in the binary representation of (k-1).

Here's the elegant solution: The answer is 'a' + (number of 1's in binary representation of (k-1)).

```python
# Time: O(log k) | Space: O(1)
def kthCharacter(k):
    """
    Returns the k-th character (1-indexed) in the infinite string.

    The key insight: Each time we take the "right half" in the binary
    tree representation, we increment the character by 1.
    The number of right-half traversals equals the number of 1's
    in the binary representation of (k-1).
    """
    # Start with base character 'a'
    base_char = 'a'

    # We need to count how many times we increment from 'a'
    # This equals the number of 1's in binary representation of (k-1)
    # Why (k-1)? Because position 1 corresponds to 0 increments.

    # Convert to 0-indexed for easier bit manipulation
    zero_indexed = k - 1

    # Count number of 1 bits (popcount/hamming weight)
    # Each 1 bit represents taking the right child in the binary tree,
    # which means incrementing the character
    increment_count = bin(zero_indexed).count('1')

    # Apply the increments to 'a'
    result_char = chr(ord(base_char) + increment_count)

    return result_char

# Alternative implementation using bit manipulation
def kthCharacterBitManipulation(k):
    """Same solution using bit manipulation instead of string conversion."""
    # Start with 'a'
    result = ord('a')

    # Work with 0-indexed position
    pos = k - 1

    # Count set bits (1's) in pos
    while pos > 0:
        # Add 1 if current bit is 1
        result += pos & 1
        # Shift right to check next bit
        pos >>= 1

    return chr(result)
```

  let incrementCount = 0;

    // Check if least significant bit is 1
    incrementCount += zeroIndexed & 1;
    // Shift right to check next bit
    zeroIndexed >>= 1;

  // Apply the increments to 'a'
  let resultCharCode = baseCharCode + incrementCount;
  return String.fromCharCode(resultCharCode);
}

function kthCharacterBuiltin(k) {
  let baseCharCode = "a".charCodeAt(0);

  let binaryStr = zeroIndexed.toString(2);
  let incrementCount = 0;
  for (let char of binaryStr) {
    if (char === "1") incrementCount++;
  }

  return String.fromCharCode(baseCharCode + incrementCount);

## Complexity Analysis

**Time Complexity: O(log k)**

- We need to examine each bit in the binary representation of (k-1)
- The number of bits is ⌊log₂(k)⌋ + 1, which is O(log k)
- In the bit manipulation approach, we shift bits right until we reach 0
- In the string conversion approach, we convert to binary string of length O(log k)

**Space Complexity: O(1)**

- We only use a constant amount of extra space for variables
- Even when converting to binary string, the string length is O(log k), but this is typically considered O(1) for complexity analysis unless specified otherwise

## Common Mistakes

1. **Off-by-one errors with indexing**: The problem uses 1-indexed positions, but bit manipulation is naturally 0-indexed. Forgetting to convert `k` to `k-1` will give wrong results. Always test with k=1 (should return 'a') and k=2 (should return 'b').

2. **Simulating the process for large k**: As discussed, the brute force approach fails for large k due to exponential growth. Candidates might try to optimize by only keeping track of the relevant portion, but even that requires understanding the binary structure.

3. **Incorrect handling of character overflow**: While the problem constraints ensure we won't go past 'z', some candidates worry about 'z' → 'a' wrapping. The mathematical solution naturally handles this since we're only counting increments up to 25 (for k ≤ 2^25).

4. **Misunderstanding the binary relationship**: Some candidates try to find the largest power of 2 less than k and use that directly, but forget that we need to count ALL 1's in the binary representation, not just determine if k is a power of 2.

## When You'll See This Pattern

This problem teaches the important pattern of **recognizing binary tree or divide-and-conquer structures in seemingly sequential processes**. When you see a process that repeatedly applies an operation to generate new data, especially with exponential growth, consider:

1. **Does it have a self-similar or fractal structure?** Like this problem where the second half depends on the first half.
2. **Can positions be mapped to paths in a binary tree?** Each decision (left/right) corresponds to a bit in a binary representation.

## Key Takeaways

1. **Exponential growth often implies logarithmic solutions**: When a process doubles at each step, look for patterns that allow you to compute results in O(log n) time by understanding the binary structure.

2. **Binary representations encode tree traversals**: The bits of (k-1) often represent the path from the root to the k-th element in a complete binary tree. A '0' bit might mean "go left" and a '1' bit might mean "go right" (or vice versa).

3. **Test small cases to discover patterns**: By working through k=1 through k=8 manually, you can often spot the relationship between k and the answer. The sequence for this problem is: a, b, b, c, b, c, c, d... which reveals the binary counting pattern.
