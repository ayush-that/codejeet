# How to Solve Ugly Number II

An **ugly number** is a positive integer whose prime factors are limited to 2, 3, and 5. Given an integer `n`, return the `nth` ugly number. This problem is tricky because we need to generate ugly numbers in order, and a naive approach checking every integer would be far too slow for typical constraints (n up to 1690). The key insight is that each ugly number can be generated by multiplying a smaller ugly number by 2, 3, or 5.

## Visual Walkthrough

Let's trace through finding the first 10 ugly numbers step by step:

We know the first ugly number is 1. To generate subsequent numbers, we maintain three pointers (or indices) that track the smallest ugly numbers we haven't yet multiplied by 2, 3, and 5 respectively.

**Initial state:**

- `ugly = [1]`
- `i2 = i3 = i5 = 0` (all pointing to the first ugly number, 1)

**Step 1:**

- Candidate for next ugly number:
  - `next2 = ugly[i2] * 2 = 1 * 2 = 2`
  - `next3 = ugly[i3] * 3 = 1 * 3 = 3`
  - `next5 = ugly[i5] * 5 = 1 * 5 = 5`
- Minimum is 2 → add to list: `ugly = [1, 2]`
- Since 2 came from multiplying by 2, increment `i2 = 1`

**Step 2:**

- Candidates:
  - `next2 = ugly[1] * 2 = 2 * 2 = 4`
  - `next3 = ugly[0] * 3 = 1 * 3 = 3`
  - `next5 = ugly[0] * 5 = 1 * 5 = 5`
- Minimum is 3 → add to list: `ugly = [1, 2, 3]`
- Increment `i3 = 1`

**Step 3:**

- Candidates:
  - `next2 = ugly[1] * 2 = 2 * 2 = 4`
  - `next3 = ugly[1] * 3 = 2 * 3 = 6`
  - `next5 = ugly[0] * 5 = 1 * 5 = 5`
- Minimum is 4 → add to list: `ugly = [1, 2, 3, 4]`
- Increment `i2 = 2`

Continue this process, and after 9 steps we'll have: `[1, 2, 3, 4, 5, 6, 8, 9, 10, 12]`

Notice that 7 and 11 are not in the list because they have prime factors other than 2, 3, and 5.

## Brute Force Approach

A naive approach would be to check every integer starting from 1, determine if it's ugly (by repeatedly dividing by 2, 3, and 5 until we reach 1 or find another prime factor), and count until we reach the nth ugly number.

# Time: O(n * log(n)) but with huge constant factor | Space: O(1)
def is_ugly(num):
    """Helper function to check if a number is ugly"""
    if num <= 0:
        return False
    for factor in [2, 3, 5]:
        while num % factor == 0:
            num //= factor
    return num == 1

def nthUglyNumber_brute(n):
    count = 0
    num = 1
    while count < n:
        if is_ugly(num):
            count += 1
        num += 1
    return num - 1

**Why this fails:** This approach is extremely inefficient. For n=1690 (the typical upper bound), the 1690th ugly number is 2123366400. We'd need to check over 2 billion numbers, and for each, perform division operations. The time complexity is roughly O(n log n) but with such a large constant factor that it's impractical for interview constraints.

## Optimized Approach

The key insight is that **every ugly number (except 1) can be generated by multiplying a smaller ugly number by 2, 3, or 5**. This means we can build the sequence dynamically:

1. Start with `ugly = [1]`
2. Maintain three pointers `i2`, `i3`, `i5` that track the smallest ugly numbers we haven't yet multiplied by 2, 3, and 5 respectively
3. At each step, compute:
   - `next2 = ugly[i2] * 2`
   - `next3 = ugly[i3] * 3`
   - `next5 = ugly[i5] * 5`
4. The next ugly number is the minimum of these three
5. Add it to the list and increment the pointer(s) that produced this minimum (if multiple produce the same value, increment all of them to avoid duplicates)

This is essentially merging three sorted sequences:

- Sequence 1: 1×2, 2×2, 3×2, 4×2, 5×2, 6×2, 8×2, ...
- Sequence 2: 1×3, 2×3, 3×3, 4×3, 5×3, 6×3, 8×3, ...
- Sequence 3: 1×5, 2×5, 3×5, 4×5, 5×5, 6×5, 8×5, ...

We're picking the smallest available number from the "front" of each sequence at each step.

## Optimal Solution

Here's the complete implementation using dynamic programming with three pointers:

<div class="code-group">

```python
# Time: O(n) | Space: O(n)
def nthUglyNumber(n):
    """
    Returns the nth ugly number using the three-pointer approach.

    The key insight: every ugly number (except 1) is a smaller ugly number
    multiplied by 2, 3, or 5. We maintain three pointers to track which
    ugly number to multiply next for each factor.
    """
    # Base case: first ugly number is 1
    if n <= 0:
        return 0

    # Initialize DP array to store ugly numbers
    ugly = [0] * n
    ugly[0] = 1  # First ugly number

    # Pointers for multiples of 2, 3, and 5
    i2 = i3 = i5 = 0

    # Generate ugly numbers from index 1 to n-1
    for i in range(1, n):
        # Calculate next candidates by multiplying with factors
        next2 = ugly[i2] * 2
        next3 = ugly[i3] * 3
        next5 = ugly[i5] * 5

        # The next ugly number is the minimum of the three candidates
        next_ugly = min(next2, next3, next5)
        ugly[i] = next_ugly

        # Increment pointer(s) that produced the minimum
        # Use separate if statements (not elif) to handle duplicates
        # e.g., 6 = 2*3 = 3*2, we need to increment both i2 and i3
        if next_ugly == next2:
            i2 += 1
        if next_ugly == next3:
            i3 += 1
        if next_ugly == next5:
            i5 += 1

    # Return the nth ugly number (0-indexed array)
    return ugly[n - 1]
```

## Complexity Analysis

**Time Complexity: O(n)**

- We perform exactly n-1 iterations (for i from 1 to n-1)
- Each iteration does constant work: three multiplications, one min operation, and up to three pointer increments
- No nested loops, so linear time overall

**Space Complexity: O(n)**

- We store all n ugly numbers in an array
- The three pointers use constant extra space
- Could be optimized to O(1) if we only needed the nth ugly number and not all previous ones, but we need the previous numbers to generate new ones

## Common Mistakes

1. **Using `else if` instead of separate `if` statements for pointer increments**: When a number can be generated in multiple ways (e.g., 6 = 2×3 = 3×2), you need to increment all relevant pointers. Using `else if` would only increment one, leading to duplicates in the sequence.

2. **Off-by-one errors with array indexing**: Remember that arrays are 0-indexed but n is 1-indexed. The nth ugly number is at position n-1 in the array. Also, make sure to initialize `ugly[0] = 1` before the loop.

3. **Not handling the base case n ≤ 0**: While the problem states n is positive, it's good practice to handle edge cases. Return 0 or throw an appropriate exception.

4. **Starting pointers at wrong values**: The pointers should start at 0, not 1, because the first ugly number (1) needs to be multiplied by each factor to generate the first candidates (2, 3, 5).

## When You'll See This Pattern

This "multiple pointers merging sorted sequences" pattern appears in several other problems:

1. **Merge k Sorted Lists (Hard)**: Instead of three sequences from multiplying by 2, 3, 5, you have k sequences from k linked lists. The min-heap approach for that problem is analogous to our three-pointer approach here.

2. **Super Ugly Number (Medium)**: This is a direct generalization where instead of fixed factors [2, 3, 5], you're given an array of primes. The solution uses the same pattern but with k pointers instead of three.

3. **Find K Pairs with Smallest Sums (Medium)**: You generate pairs from two arrays in sorted order, similar to how we generate ugly numbers from three sequences.

The core idea is recognizing when you need to generate elements in sorted order from multiple sources, and using pointers (or a heap) to always pick the next smallest element.

## Key Takeaways

1. **Dynamic generation beats checking**: When you need to generate numbers with specific properties in order, it's often more efficient to generate them directly using known relationships rather than checking each number.

2. **Multiple sequences merging**: When each new element can be derived from previous elements in multiple ways (multiplying by different factors), think of maintaining pointers to track the "front" of each sequence.

3. **Handle duplicates carefully**: When an element can be generated in multiple ways, ensure your logic increments all relevant pointers to avoid duplicates in the output sequence.
